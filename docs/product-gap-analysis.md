# TripSplit 產品缺口與開發討論文件

## 文件目的

這份文件用來整理目前專案的產品缺口，作為後續功能規劃、優先級排序與開發討論的基礎。

重點不是列出所有「可以做的功能」，而是從「這個產品要真的拿給多人使用，還缺什麼」的角度來盤點。

> 狀態說明：
>
> - **已完成**：目前專案中已有對應資料模型、API 或 UI 流程可使用
> - **部分完成**：已有部分基礎，但尚未達到文件原本描述的完整程度
> - **尚未完成**：目前專案中尚未看到對應實作

---

## 目前已具備的核心能力

依照目前專案狀態，已經具備以下核心流程：

- 建立旅程
- 加入旅伴 / 成員管理
- 新增旅程費用
- 多種分帳方式
- 匯率換算
- 收據上傳
- 自動結算
- 支出統計
- 結算品項明細顯示

也就是說，產品已經有「可用的分帳骨架」，但若要進一步成為可以穩定提供多人使用的服務，還有幾個明顯缺口。

---

## 功能缺口分級

以下將缺口分成四個層級，方便後續討論時決定先補哪一類。

---

## 第一層：最優先補齊的缺口

這一層的功能，會直接影響產品是否能夠真正上線給多人使用。

### 1. 使用者身份識別 / 登入系統（已完成）

此項已完成，系統現在已能辨識每位使用者的身份，並把旅程、成員、費用與付款狀態綁定到對應帳號。

目前狀態：

- 已有登入頁面與 session API
- 已有 `User`、`Session` 資料模型
- 首頁、建立旅程頁、旅程詳情頁都已依登入狀態調整流程

### 2. 權限控制（已完成）

此項已完成，系統已針對旅程、成員、費用與付款相關操作建立權限邏輯。

目前狀態：

- 已引入 `Trip.ownerId`、`Member.userId`、`Expense.createdById`
- 已在 trip / member / expense / payment API 中加上權限檢查
- UI 已依權限顯示可否刪除旅程、管理成員、編輯/刪除費用

### 3. 編輯既有費用（已完成）

此項已完成，目前費用可在既有表單流程中進行修改，不再只有新增與刪除兩種操作。

目前狀態：

- 費用列表已可進入編輯模式
- 可回填既有費用到表單並送出 PATCH 更新
- 可修改金額、品項、付款人、分攤方式、匯率、收據與備註

### 4. 更完整的錯誤處理（部分完成）

需要處理並清楚回饋使用者以下情況：

- 匯率抓取失敗
- 收據上傳失敗
- 網路中斷
- 儲存失敗
- 資料格式不正確

若錯誤處理不足，產品雖然能跑，但不夠可靠。

目前狀態：

- 已有網路錯誤、重試與 API 錯誤訊息提示
- 已有匯率查詢失敗與收據上傳失敗的 UI 回饋
- 但目前仍缺少更細緻的欄位級驗證、流程級恢復與統一錯誤體驗設計

---

## 第二層：會大幅提升實用性的功能

這一層會讓產品從「有做出來」進一步變成「真的好用」。

### 1. 結算完成 / 已付款狀態（已完成）

此項已完成，結算不再只是建議金額，已可被標記為已付款並保留狀態紀錄。

目前狀態：

- 已有 `SettlementPayment` 資料模型
- 可標記付款完成、撤銷與恢復付款狀態
- 已付款紀錄會影響待結算結果

### 2. 依人查看明細（已完成）

此項已完成，除了依付款關係查看外，也已支援從個人視角查看待付、待收與對應品項。

目前狀態：

- 已新增「依人查看結算明細」區塊
- 可從每位成員角度看到待付、待收與對應品項

### 3. 搜尋與篩選（已完成）

此項已完成，目前費用列表已可依多個條件快速縮小範圍。

目前狀態：

- 已支援關鍵字、類別、付款人、日期區間搜尋與篩選

### 4. 匯出結算結果（已完成）

此項已完成第一階段需求，目前已能把結算或費用資料匯出成可分享或備份的檔案格式。

目前狀態：

- 已支援文字版結算明細匯出
- 已支援 JSON 備份匯出
- 已支援 CSV 費用匯出

尚未完成的部分：

- 圖片版結算卡
- PDF / 列印版輸出

### 5. 結算備註 / 特殊處理註記（部分完成）

真實分帳常有例外狀況，例如：

- 這筆先不算
- 這筆私人支出不分帳
- 這筆已經線下處理

若沒有這類註記能力，系統結果雖正確，實務上仍會不好用。

目前狀態：

- 標記付款時已可輸入付款備註
- 費用已可設定「保留記帳但不納入結算」與「線下處理 / 私人支出」
- 結算頁也會明確列出這些不納入自動結算的費用
- 但更完整的特殊規則管理（例如批次規則、更多例外類型）仍未完成

---

## 第三層：產品成熟度相關缺口

這一層不一定要最先做，但會影響產品是否值得信任。

### 1. 操作紀錄 / 歷史紀錄（已完成）

此項已完成，目前已能追蹤成員、費用與付款相關的主要操作紀錄。

目前狀態：

- 已有 `ActivityLog` 資料模型
- 成員、費用、付款狀態變更都會記錄
- UI 已有活動紀錄頁籤可查看最近紀錄

### 2. 資料備份與還原（部分完成）

若要正式上線，資料可靠性一定要補強。

需要考慮：

- 定期備份
- 異常還原
- 旅程資料匯出
- 部署環境資料保存策略

尤其目前若仍以 SQLite 為主，這一塊更需要提前規劃。

目前狀態：

- 已有 JSON / CSV 匯出能力
- 已支援從 JSON 備份匯入並建立新的還原旅程
- 但尚未提供定期備份、自動備份與真正的保存策略

### 3. 通知機制（部分完成）

未來可以思考的通知情境：

- 有人新增費用
- 有新成員加入旅程
- 結算結果更新
- 某筆款項被標記為已付款

通知能提升多人協作感，但不一定要第一時間開發。

目前狀態：

- 已有站內通知中心
- 成員異動、費用新增/修改/刪除、付款狀態更新等事件會產生通知
- 但尚未提供 email、push 或更完整的通知偏好設定

### 4. 手機操作體驗優化（部分完成）

旅遊分帳工具很可能主要在手機上使用。

因此需要進一步檢查：

- 表單輸入長度是否過長
- 小螢幕閱讀是否清楚
- 單手操作是否方便
- 收據上傳流程是否順手

這屬於成熟化與體驗優化的一部分。

目前狀態：

- 已有基本 responsive layout、手機可滾動 tab、較大的互動區域與相機收據上傳
- 已新增手機版底部快速切換列，方便在消費 / 記帳 / 結算 / 總結之間切換
- 但仍可進一步優化更深層的 mobile UX 細節

---

## 第四層：加分型功能

這一層不一定是 MVP 必需，但會讓產品更有差異化與完整感。

### 1. 收據 OCR（尚未完成）

支援從收據影像中辨識：

- 金額
- 日期
- 店名

可大幅降低手動輸入成本。

目前狀態：

- 目前僅支援收據上傳，尚未看到 OCR 辨識流程

### 2. 更完整的多語系 / 多幣別體驗（部分完成）

目前已有匯率能力，但若要走得更完整，可以補：

- 多語系介面
- 不同幣別顯示邏輯
- 更清楚的換匯資訊呈現

目前狀態：

- 已支援多幣別記帳與匯率轉換
- 但尚未看到多語系架構與完整 locale 切換

### 3. 自訂消費類別（尚未完成）

讓使用者可以新增或調整分類，例如：

- 伴手禮
- 門票
- 租車
- 私人支出

這能讓統計與輸入更貼近真實使用習慣。

目前狀態：

- 類別仍固定定義在 `src/lib/constants.ts`
- 尚未支援使用者自訂類別

### 4. 旅程總結頁（已完成）

在旅程結束後提供整體摘要，例如：

- 誰支付最多
- 哪天花費最高
- 主要支出類別
- 單筆最大消費

這類功能對使用者很有「結束旅程後回顧」的價值。

目前狀態：

- 已新增旅程總結檢視
- 可查看總支出、待結算筆數、特殊處理費用、支付最多成員、最大單筆消費、最高支出日與最大類別

---

## 目前建議優先順序（Top 5）

如果只先選 5 個最值得優先投入的方向，建議排序如下：

1. 更完整的錯誤處理
2. 資料備份與還原
3. 通知機制
4. 手機操作體驗優化
5. 收據 OCR / 自訂消費類別

這個排序的理由是：這些才是目前仍會影響產品從「已具備核心流程」走向「可長期穩定使用」的主要缺口。

---

## 建議的後續討論方式

後續討論可以依照下面順序進行：

### 第一階段：確認 MVP 上線前必補項目

建議先回答：

- 這個產品短期是要 demo、封閉測試，還是真的提供多人使用？
- 有沒有登入需求？
- 是否允許任何旅伴編輯所有資料？
- 結算完成狀態是否為必要？

### 第二階段：確認資料模型是否要調整

例如：

- 是否要引入 User 模型
- Trip / Member / User 的關係要怎麼設計
- 是否要記錄操作歷史
- 是否要為結算建立付款狀態資料

### 第三階段：決定實作優先級

建議可以分成：

- 上線前必做
- 近期應做
- 中長期加分項

---

## 結論

TripSplit 目前已具備完整的旅遊分帳基本流程，而且身份、權限、費用編輯、結算付款狀態、依人查看與匯出等核心缺口都已補上。接下來若要往真正可上線、可多人長期使用的方向前進，最需要優先補齊的是：

- 更完整的錯誤處理
- 備份 / 還原與資料可靠性
- 特殊結算規則與備註流程
- 通知、手機體驗與產品成熟度細節

這份文件的用途不是一次定案，而是作為後續功能拆分與規劃時的共同基準。
